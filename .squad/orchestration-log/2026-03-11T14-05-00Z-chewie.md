## Deliverable Summary

### Module Architecture Created
```
js/
├── config.js              # Constants, maze data, colors (114 lines)
├── main.js                # Entry point (13 lines)
├── engine/
│   ├── audio.js           # SoundManager class (166 lines)
│   └── renderer.js        # Sprites static class (720 lines)
└── game-logic.js          # Game class (791 lines)
```

**Total:** 5 modules, 1804 lines (vs. 1789 monolith → 15-line header overhead)

### Key Architecture Decisions

1. **Config as Pure Data Foundation**
   - Zero dependencies, consumed by all modules
   - 114 lines: constants, maze grid (31×28), ghost configs, Simpsons color palette, game states, direction vectors
   - Enables parallel work on rendering and game logic

2. **Engine Separation (Audio + Renderer)**
   - **audio.js:** SoundManager class with Web Audio API lifecycle
     - Simpsons theme jingle, D'oh death sound, Duff power-up, background loop
   - **renderer.js:** Sprites static class with stateless methods
     - Homer with hair/eyes/stubble rendering
     - 4 ghost characters (Burns, Bob, Nelson, Snake) with consistent pattern
     - Donut, Duff, maze rendering at 720 lines (cohesive, single responsibility)

3. **Game Logic as Orchestrator**
   - **game-logic.js:** Game class (791 lines)
     - Depends on all modules
     - Game loop, state machine (7 states), player movement + collision
     - Ghost AI with scatter/chase/frightened modes
     - Scoring and level progression

4. **Load Strategy: Plain `<script>` Tags (No Bundler)**
   - Dependency flow: config → audio → renderer → game-logic → main
   - Explicit load order in index.html
   - Rationale: Vanilla JS project, simple DAG, low development friction, global namespace intentional

### Module Load Order Rationale

```
config.js (no deps)
  ↓
audio.js (uses config constants)
  ↓
renderer.js (uses config constants, COLORS)
  ↓
game-logic.js (uses all modules)
  ↓
main.js (instantiates Game)
```

**Why This Order Matters:**
- Config must load first (no dependencies, pure data)
- Audio and renderer can load in any order after config (no cross-dependency)
- Game logic loads after both (depends on both)
- Main loads last (thin entry point)

## Alternatives Considered & Rejected

1. **ES Modules with `type="module"`** — Rejected because it requires export/import syntax changes, relative path handling complexity. Current global approach simpler for vanilla JS.

2. **Split Renderer by Entity Type** (player.js, ghosts.js, maze.js, particles.js) — Rejected because drawing code is tightly coupled to config constants; pattern consistency across 4 ghost renderers is clearer in one file.

3. **Bundle with Webpack/Vite** — Rejected to keep zero-build-step simplicity. Browser handles 5 small files efficiently; no need for build complexity.
