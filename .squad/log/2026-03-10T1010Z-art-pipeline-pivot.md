## Problem Statement

**V2 Frames Failure (2026-03-10T09:27Z)**

Founder rejected v2 animation frames generated by Kontext Pro. Boba's audit confirmed critical consistency failure: V2 scored 7/40 vs V1's 39/40. Each frame looks like a different character:
- Hair color: random per frame (pink → cyan → purple → silver)
- Outfit color: random per frame (red gi → green gi → brown gi)
- Body type: inconsistent proportions frame-to-frame
- Energy VFX: random appearance/color

**Root Cause:** Kontext Pro was used as text-to-image generator ("generate frame 2 with anticipation pose"). No reference anchoring per frame. Model had zero constraint on visual consistency.

**Business Impact:** 56 frames generated, zero frames usable. Sprite pipeline bottleneck threatens Sprint 2 timeline.

## Research & Analysis

### Solo's Comprehensive Survey

Conducted 9 independent web searches covering:
- FLUX model variants (FLUX 1.1 Pro, FLUX.2 Dev, FLUX.2 Flex)
- ControlNet architecture & conditioning modes
- Academic approaches (Sprite Sheet Diffusion, LATENTMAN)
- SaaS tools (PixelLab, Spritesheets.AI, AutoSprite, Scenario)
- Indie game pipelines (Donkey Kong Country, Killer Instinct, modern fighting games)
- Professional tools (SpriteStar, Spritesheet Renderer, ComfyUI)

### Five Approaches Evaluated

**Approach A: ComfyUI + FLUX.2 Dev + ControlNet Pose + IP-Adapter**
- Master sprite → pose skeletons → ControlNet (pose) + IP-Adapter (appearance) → conditioned frame generation
- Consistency: 8/10 (high, 20% variation in fine details)
- Complexity: HIGH (local setup, model downloads ~20GB, GPU 12-24GB VRAM)
- Cost: GPU time (cloud rental or local)
- Verdict: Right direction, requires significant engineering. Our current API-only FLUX models (1.1 Pro, Kontext Pro) cannot do this — needs FLUX.2 Dev + open weights
- ChatGPT research validation: ✅ Correctly identified solution direction ❌ Oversimplified implementation (assumed Kontext Pro could accept skeleton images directly)

**Approach B: PixelLab (Web-based SaaS)**
- Purpose-built AI tool for pixel art game assets
- Skeleton-based animation system with style consistency via reference image
- Consistency: 8/10 (for pixel art style, purpose-built)
- Complexity: LOW (web tool, no setup)
- Cost: SaaS subscription
- Verdict: Excellent IF our art direction is pixel art. Quick spike worth running (30 min evaluation). NOT suitable if Ashfall goes cel-shaded or higher-res.

**Approach C: 3D-to-2D Pipeline (Mixamo + Blender)**
- Free 3D models (Mixamo library) + fighting animations (mocap mocap library)
- Blender orthographic render (side view) → PNG export (512×512 transparent)
- Cel-shade toon shader (Shader-to-RGB + ColorRamp) for 2D fighting game look
- Consistency: PERFECT 10/10 (same mesh = perfect consistency by definition)
- Complexity: MEDIUM (one-time Blender setup, then fast batch rendering)
- Cost: FREE (Blender + Mixamo are free; optional SpriteStar plugin $10)
- Verdict: **Most reliable. Zero AI randomness. Proven technique used by professional studios.** Classic pipeline for Donkey Kong Country, Killer Instinct, modern indie fighters.
- **Status:** Spike complete (Chewie + Coordinator validated with 4 animations = 95 frames)

**Approach D: Sprite Sheet Diffusion (Academic Research)**
- arXiv:2412.03685 — generates entire sprite sheets in one diffusion pass
- Consistency: 7/10 (moderate-high based on paper metrics)
- Complexity: HIGH (research code, requires ML engineering)
- Verdict: Interesting for future. Not production-ready today.

**Approach E: SaaS Sprite Tools (Spritesheets.AI, AutoSprite, Scenario)**
- One-click generation with atlas/metadata exports
- Consistency: 6/10 (moderate, less control than ControlNet)
- Complexity: VERY LOW (upload image, get sheet)
- Verdict: Good for prototyping. Insufficient for production fighting game sprites where frame precision matters.

### Ranking: Top 3 by Feasibility

**🥇 RANK 1: 3D-to-2D Pipeline (Mixamo + Blender)**
- Time to first result: 2-4 hours
- Cost: FREE (Blender + Mixamo + SpriteStar plugin ~$10)
- Risk: Medium (art style gap needs post-processing)
- Recommended path: Adopt as primary pipeline
- Why first: Guaranteed consistency, zero AI randomness, proven in production studios

**🥈 RANK 2: ComfyUI + FLUX.2 Dev + ControlNet + IP-Adapter**
- Time to first result: 1-2 days
- Cost: GPU time (cloud rental)
- Risk: HIGH (complex pipeline, requires 12-24GB VRAM, ~20% variation still needs QA)
- Recommended path: If we still want AI, setup locally with engineering support
- Why second: Best AI approach if we pursue it, but requires significant infrastructure investment

**🥉 RANK 3: PixelLab (SaaS, pixel art only)**
- Time to first result: 30 minutes
- Cost: SaaS subscription
- Risk: MEDIUM (vendor lock-in, pixel art style only)
- Recommended path: Quick spike test if art direction confirms pixel art
- Why third: Purpose-built tool, but style-constrained

## Knowledge Capture

### For Future Projects

1. **Text-only AI is unsuitable for animation.** Require structural conditioning (ControlNet) + appearance conditioning (IP-Adapter) or deterministic rendering.
2. **3D-to-2D is the proven professional pipeline.** Donkey Kong Country, Killer Instinct, modern indie fighters use this approach. Not a compromise — it's the standard.
3. **ControlNet is the key innovation for AI sprites.** If pursuing AI, FLUX.2 Dev (open weights) + ControlNet Union + IP-Adapter is the correct architecture. API-only FLUX models won't work.
4. **Consistency is non-negotiable for animation.** A consistent ugly sprite is infinitely more useful than a beautiful inconsistent one.
5. **Spike early, fail fast.** Boba's audit proved wrong approach in < 2 hours. Coordinator's render validation proved right approach in < 4 hours.
