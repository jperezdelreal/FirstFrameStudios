### 1.3 What We CAN Do

- ✅ Text-to-image generation (all 3 models)
- ✅ Image editing with reference images (Kontext Pro, FLUX 2 Pro)
- ✅ Character consistency via `input_image` reference propagation (Kontext Pro)
- ✅ Multi-reference image input, up to 4–10 images (FLUX 2 Pro)
- ✅ Seed-based reproducibility (Kontext Pro)
- ✅ Various resolutions, multiples of 32 (all models)
- ✅ High-quality output in PNG format with base64 response

### 1.4 What We CANNOT Do

- ❌ **ControlNet / OpenPose** — No pose skeleton input. Poses described via text only.
- ❌ **LoRA fine-tuning** — No custom model training on our characters.
- ❌ **IPAdapter / InstantID** — No face/body identity transfer modules.
- ❌ **Inpainting** — No selective region editing. Must regenerate full frame.
- ❌ **ComfyUI-style pipelines** — API calls only, no node-based orchestration.
- ❌ **AnimateDiff** — No temporal consistency between frames (but Kontext reference helps).
- ❌ **FLUX 2 Flex** — Not deployed on our Azure resource.

**Key upgrade from v2:** We are no longer limited to prompt anchoring alone. Kontext Pro's `input_image` parameter gives us **visual reference propagation** — a massive improvement over the previous text-only consistency approach.

## 2. Recommended Workflow — Multi-Model Pipeline

### 2.1 Three-Model Strategy

Each model has a specific role in our pipeline:

```
┌──────────────────┐     ┌──────────────────────┐     ┌────────────────────┐
│  FLUX 2 Pro      │     │  Kontext Pro          │     │  FLUX 1.1 Pro      │
│  4 req/min       │────▶│  30 req/min           │     │  30 req/min        │
│                  │     │                       │     │                    │
│  Hero frames     │     │  All gameplay sprites │     │  Backgrounds,      │
│  1024×1024       │     │  512×512              │     │  props, VFX        │
│  2 per character │     │  ~1,020 frames total  │     │  512×512           │
└──────────────────┘     └──────────────────────┘     └────────────────────┘
       ▲                          ▲
       │                          │
   Boba approves              Uses hero frame
   hero frame                 as input_image
```

**Step 1 — Hero Frames (FLUX 2 Pro):**
Generate ONE perfect idle pose per character at 1024×1024. This is the canonical reference image that defines the character's visual identity. Use FLUX 2 Pro for maximum quality (4.0 MP ceiling). Boba reviews and approves before any further generation proceeds. Generate P1 and P2 palette versions.

**Step 2 — Pose Production (Kontext Pro):**
With the approved hero frame as `input_image`, generate every gameplay pose at 512×512. Kontext Pro maintains character identity from the reference image while following pose instructions in the prompt. Use the same `seed` value within a pose sequence for frame-to-frame consistency. This is where ~1,020 frames get produced.

**Step 3 — Non-Character Assets (FLUX 1.1 Pro):**
Backgrounds, projectiles, ember particles, hit sparks, UI elements — anything that doesn't need character consistency. Fast, no reference image needed.

### 2.2 Production Time Estimates

**Hero Frames (FLUX 2 Pro at 4 req/min):**

| Task | Frames | Time |
|---|---|---|
| Kael hero (P1 palette) | 3–5 attempts to get perfect frame | 1–2 min |
| Kael hero (P2 palette) | 3–5 attempts | 1–2 min |
| Rhena hero (P1 palette) | 3–5 attempts | 1–2 min |
| Rhena hero (P2 palette) | 3–5 attempts | 1–2 min |
| **Total hero frames** | ~12–20 generations | **~5–10 min** |

**Gameplay Sprites (Kontext Pro at 30 req/min):**

| Phase | Pose Sets | Frames (est.) | Generation Time | With 25% regen |
|---|---|---|---|---|
| P0 (PoC) | 3 poses × 2 chars = 6 sets | ~56 frames | ~2 min | ~2.5 min |
| P1 (MVP) | 34 poses × 2 chars = 68 sets | ~680 frames | ~23 min | ~28 min |
| P2 (Polish) | 14 poses × 2 chars = 28 sets | ~284 frames | ~9.5 min | ~12 min |
| **Total** | **51 poses × 2 chars = 102 sets** | **~1,020 frames** | **~34 min** | **~42 min** |

**Raw generation is fast.** The bottleneck is review and iteration, not API throughput. Budget 2–3× generation time for the full review cycle: ~2 hours for all gameplay sprites including QA passes.

### 2.3 Consistency Strategy (Upgraded)

The v2 brief identified character consistency as the #1 risk and rated our mitigation as "weakest" (prompt anchoring only). With Kontext Pro, we now have a **proven visual reference pipeline:**

1. **Hero frame as identity anchor** — Every Kontext Pro call includes the approved hero frame as `input_image`. The model sees what the character looks like, not just text describing them.
2. **Seed control for sequence coherence** — Lock a seed per pose sequence (e.g., `seed: 1001` for all `idle` frames). Same seed + similar prompt = more consistent output.
3. **Prompt anchoring reinforcement** — Still include full character description in every prompt, but now as a secondary consistency layer rather than the only one.
4. **Post-processing recolor** — Programmatic palette enforcement for P2 polish (Section 6.5).

**Risk reassessment vs v2:**

| Risk | v2 Severity | v3 Severity | Reason |
|---|---|---|---|
| Proportions drift between frames | High | **Medium** | Kontext reference constrains proportions |
| Costume details vanish | Medium | **Low** | Reference image preserves visual details |
| Color palette drift | Low | **Low** | Unchanged — recolor script handles this |
| Animation shimmer at 60 FPS | High | **Medium** | Same-seed sequences reduce frame variance |
| Hair/scar inconsistency | Medium | **Low** | Reference image preserves these features |

## 3. Sprite Specifications

### 3.1 Canvas Dimensions

**512×512 px** — Joaquín's preference, validated as industry sweet spot for AI sprite generation.

- FLUX requires multiples of 32. 512 satisfies this.
- In-game characters render at ~30×60 px (1280×720 viewport). 512 source provides 8.5× safety margin.
- Power-of-2 texture size — optimal for GPU memory and mipmapping.
- File size: ~50–100 KB per PNG. ~1,020 sprites ≈ 50–100 MB total — within Godot norms.

**Hero frames exception:** Generated at 1024×1024 via FLUX 2 Pro, then downscaled to 512×512 using nearest-neighbor interpolation. Higher initial resolution gives FLUX more working space for detail.

### 3.2 Format

- **File format:** PNG with alpha transparency
- **Color depth:** 32-bit RGBA (8 bits per channel)
- **Alpha handling:** Full alpha channel — NO pre-multiplied alpha (Godot expects straight alpha)
- **Outline approach:** 1–2 px dark outline INSIDE the character silhouette. Kael: `#1F1A14`. Rhena: `#1A0F0A`.
- **Background:** Fully transparent (alpha = 0)

### 3.3 Orientation

- **Facing direction:** All sprites face RIGHT
- **Mirroring:** Engine handles flip via `scale.x = -1.0` (already implemented)
- **Reason:** 50% sprite count reduction. Standard fighting game practice.

### 3.4 Ground Line / Origin Point

- **Origin (0, 0):** Center-bottom of canvas (`x=256`, `y=512`)
- **Ground line:** Y = 512 (bottom edge). Feet touch this line in standing poses.
- **Godot import:** `sprite.offset = Vector2(0, -256)` — feet at node origin.

### 3.5 Padding / Safe Zone

- **Active area:** 440×480 px within the 512×512 canvas
  - 36 px horizontal margin (camera shake tolerance)
  - 32 px top margin (overhead attacks, VFX)
  - 0 px bottom margin (feet on ground line)
- **Attack extensions:** Limbs may break safe zone by up to 80 px horizontally during active frames.

## 5. Animation Pose Catalog

### Format

Each pose entry includes:
- **Pose name** (matches `CharacterSprite.pose` property)
- **Visual description** (what the character looks like)
- **Frame count** (60 FPS timing from ART-DIRECTION.md)
- **Key frames** (critical frames for animation clarity)
- **Priority** (P0 = PoC, P1 = MVP, P2 = polish)

**Timing reference:** 1 frame = 16.67ms at 60 FPS.

---

### 5.1 Core Stances (Looping)

#### `idle`
- **Description:** Guard stance. Kael: upright, hands near chin level, elbows tucked, weight centered. Rhena: wide stance, knees bent, fists clenched at sides, forward lean.
- **Frame count:** 8 frames (looping)
- **Key frames:** 
  - Frame 1: neutral pose
  - Frame 4: peak inhale (chest expands 2px, shoulders rise 1px)
  - Frame 8: return to neutral (exhale complete)
- **Priority:** **P0** (needed for PoC)
- **Notes:** Breathing cycle is subtle. Kael's head stays level. Rhena's head bobs slightly with breathing (more aggressive).

#### `walk`
- **Description:** Forward stride. Leading leg extends forward, rear leg pushes off. Arms swing opposite legs (left arm forward = right leg forward).
- **Frame count:** 8 frames (looping)
- **Key frames:**
  - Frame 1: mid-stride (right foot forward, left arm forward)
  - Frame 4: contact (right foot plants, weight shifts)
  - Frame 5: push-off (left foot lifts)
  - Frame 8: transition to next stride (left foot forward)
- **Priority:** **P0** (needed for PoC)
- **Notes:** Kael's walk is smooth, controlled. Rhena's walk has more vertical bob (stomps).

#### `walk_2`
- **Description:** Alternate stride phase (mirrors `walk` frame 1→4 with opposite leg forward).
- **Frame count:** 8 frames (looping)
- **Key frames:** Same as `walk`, but mirrored leg lead.
- **Priority:** P1 (for smooth walk cycle transitions)

#### `crouch`
- **Description:** Low defensive stance. Knees bent, torso lowered, head level drops by ~20px. Arms guard chest/head.
- **Frame count:** 3 frames transition + 4 frames idle (7 total)
- **Key frames:**
  - Frame 1: start descent (knees begin bending)
  - Frame 3: full crouch (butt near heels, arms tight)
  - Frames 4-7: crouch idle (minimal breathing bob)
- **Priority:** P1 (defensive gameplay)

---

### 5.2 Air Movement

#### `jump_up`
- **Description:** Ascent phase. Legs tucked slightly, arms at sides or slightly raised. Head tilts back slightly.
- **Frame count:** 4 frames (non-loop)
- **Key frames:**
  - Frame 1: launch (legs compress, body coiled)
  - Frame 2: extension (legs push down, body rises)
  - Frame 4: airborne (legs relaxed, ascending)
- **Priority:** P1
- **Notes:** Kael's jump is controlled (arms stay near body). Rhena's jump is explosive (arms swing up, knees tuck higher).

#### `jump_peak`
- **Description:** Apex hang. Body at maximum height, momentary weightlessness. Arms spread slightly for balance.
- **Frame count:** 2 frames (non-loop)
- **Key frames:**
  - Frame 1: peak (body fully extended, moment of stillness)
  - Frame 2: transition to fall (slight downward tilt begins)
- **Priority:** P1

#### `jump_fall`
- **Description:** Descent phase. Legs extend downward (preparing to land), arms adjust for balance.
- **Frame count:** 4 frames (non-loop)
- **Key frames:**
  - Frame 1: start fall (legs begin extending)
  - Frame 3: pre-land (one leg extended down, arms brace)
  - Frame 4: land preparation (knees bend slightly in air)
- **Priority:** P1

---

### 5.3 Dashes

#### `dash`
- **Description:** Forward burst. Body leans forward sharply (~30° from vertical). Leading leg drives, rear leg trails. Speed lines optional.
- **Frame count:** 4 frames (non-loop)
- **Key frames:**
  - Frame 1: launch (rear leg pushes, body tilts forward)
  - Frame 3: peak speed (body horizontal lean, hair trails back)
  - Frame 4: deceleration (body begins returning upright)
- **Priority:** P1
- **Notes:** Rhena's dash is more aggressive (deeper lean, arms pumping). Kael's dash is controlled (arms stay near body).

#### `backdash`
- **Description:** Evasive retreat. Body leans backward, legs push away from opponent. Defensive posture.
- **Frame count:** 4 frames (non-loop)
- **Key frames:**
  - Frame 1: push-off (front leg drives backward)
  - Frame 2: airborne (body tilts back, feet off ground)
  - Frame 4: land (back foot touches down first, weight shifts)
- **Priority:** P1

### 5.4 Standing Attacks

#### `attack_lp` (Light Punch)
- **Description:** Quick jab. Leading hand extends forward at head height. Elbow snaps straight. Rear hand guards face.
- **Frame count:** 12 frames (4 startup + 2 active + 6 recovery)
- **Key frames:**
  - Frame 1: guard position (fist at chin)
  - Frame 4: wind-up complete (fist cocks back 4px)
  - Frame 5: **ACTIVE** — full extension (fist 40px forward from body)
  - Frame 6: **ACTIVE** — hold extension
  - Frame 12: return to guard
- **Priority:** **P0** (needed for PoC)
- **Notes:** Kael's LP is a straight punch (knuckles vertical). Rhena's LP is slightly angled down (overhand jab).

#### `attack_mp` (Medium Punch)
- **Description:** Straight power punch. Full body rotation. Rear hand drives forward, hips turn, front foot pivots.
- **Frame count:** 19 frames (6 startup + 3 active + 10 recovery)
- **Key frames:**
  - Frame 1: guard
  - Frame 6: wind-up (rear hand cocks to hip, body coils)
  - Frame 7-9: **ACTIVE** — punch extends (arm fully straight, hips rotated 45°)
  - Frame 19: return to neutral
- **Priority:** P1
- **Notes:** More body commitment than LP. Hips and shoulders rotate visibly.

#### `attack_hp` (Heavy Punch)
- **Description:** Overhand smash. Arm swings up and forward in arc. Body leans into strike. Big commitment.
- **Frame count:** 32 frames (10 startup + 4 active + 18 recovery)
- **Key frames:**
  - Frame 1: guard
  - Frame 10: wind-up (fist raised above head, body coiled back)
  - Frame 11-14: **ACTIVE** — downward arc (fist swings forward and down)
  - Frame 20: full extension (arm fully extended downward, body leaning forward)
  - Frame 32: slow return to guard
- **Priority:** P1
- **Notes:** Rhena's HP has more arc (wild haymaker). Kael's HP is more controlled (overhead strike).

#### `attack_lk` (Light Kick)
- **Description:** Shin kick. Front leg lifts and snaps forward at knee height. Quick, no wind-up.
- **Frame count:** 16 frames (5 startup + 3 active + 8 recovery)
- **Key frames:**
  - Frame 1: guard stance
  - Frame 5: knee lifts (thigh parallel to ground, shin cocked)
  - Frame 6-8: **ACTIVE** — leg extends (shin kicks forward)
  - Frame 16: foot returns to ground
- **Priority:** P1

#### `attack_mk` (Medium Kick)
- **Description:** Side kick. Rear leg chambers and thrusts forward at torso height. Hip turns.
- **Frame count:** 22 frames (7 startup + 3 active + 12 recovery)
- **Key frames:**
  - Frame 1: guard
  - Frame 7: chamber (rear knee lifts, body pivots on front foot)
  - Frame 8-10: **ACTIVE** — thrust (leg extends fully, foot horizontal)
  - Frame 22: leg retracts, foot plants
- **Priority:** P1

#### `attack_hk` (Heavy Kick)
- **Description:** Roundhouse. Rear leg swings in wide arc from outside-to-inside. Body spins. Big telegraph.
- **Frame count:** 39 frames (12 startup + 5 active + 22 recovery)
- **Key frames:**
  - Frame 1: guard
  - Frame 12: wind-up (rear leg lifts, body begins rotation)
  - Frame 13-17: **ACTIVE** — arc sweep (leg swings horizontally, body spins 180°)
  - Frame 25: leg completes arc (foot past opponent, body still rotating)
  - Frame 39: return to stance (body rotates back, leg plants)
- **Priority:** P1
- **Notes:** Rhena's HK is wild (full 360° spin possible). Kael's HK is controlled (180° max).

### 5.5 Crouching Attacks

#### `crouch_lp` (Crouching Light Punch)
- **Description:** Low jab. Torso in crouch position, front arm extends forward at waist height.

### 6.3 Seed Strategy

Seeds enable reproducible results and reduce variance within a pose sequence.

**Seed assignment scheme:**
- **Kael poses:** Seeds 1000–1999
- **Rhena poses:** Seeds 2000–2999
- **Per-pose offset:** Each pose gets a unique base seed
- **Per-frame increment:** Frames within a pose use `base_seed + frame_number`

**Example:**
```
Kael idle:      seed 1001, 1002, 1003, ..., 1008
Kael walk:      seed 1011, 1012, 1013, ..., 1018
Kael attack_lp: seed 1021, 1022, 1023, ..., 1032
Rhena idle:     seed 2001, 2002, 2003, ..., 2008
```

**Why increment per frame (not reuse)?** Same seed + different prompt = different image. Incrementing keeps seeds unique while maintaining a traceable relationship. If frame 5 of `idle` needs regeneration, re-run seed 1005 with an adjusted prompt.

### 6.7 Palette Enforcement

FLUX does not guarantee exact hex colors.

**Strategy:** Include hex codes in prompts as guidance, then apply programmatic recolor in post-processing for P2 polish.

```python
from PIL import Image
import numpy as np

TARGET_PALETTE = {
    "skin":    (0xD9, 0xB3, 0x8C),
    "hair":    (0x26, 0x1F, 0x1A),
    "gi":      (0xE0, 0xDB, 0xD1),
    "shadow":  (0xB3, 0xAD, 0xA3),
    "accent":  (0x40, 0x73, 0xD9),
    "outline": (0x1F, 0x1A, 0x14),
}

def recolor_sprite(image_path, palette, tolerance=15):
    """Replace FLUX output colors with exact palette values."""
    img = Image.open(image_path).convert("RGBA")
    data = np.array(img)
    for name, target in palette.items():
        mask = np.all(np.abs(data[:,:,:3].astype(int) - target) <= tolerance, axis=2)
        data[mask, :3] = target
    return Image.fromarray(data)
```

**Tolerance:** P0/P1: accept ±10% drift. P2: enforce exact palette via recolor.

## 7. Quality Checklist & Production Pipeline

### 7.1 Quality Checks

Every generated sprite must pass ALL checks before approval.

**Silhouette Test:** Scale to 64×64, fill solid black. Can you identify character + pose?

**Color Accuracy:** Sample 5 key colors. ±10% luminance, ±5% hue of target hex = pass.

**Pose Accuracy:** Limb positions, weight distribution, and frame timing match Section 5 descriptions.

**Frame Coherence:** Play all frames at 60 FPS. Proportions, costume details, and motion flow stay consistent.

**Style Consistency:** Outline weight (1–2 px), shading (flat), detail level all match across frames.

**Technical Compliance:**
- 512×512 px (1024×1024 for hero frames)
- PNG with alpha transparency
- Transparent background
- No blur or anti-aliasing glow
- Facing right
- Feet on ground line

### 7.2 Hero Frame Gate

**This is the most important gate in the entire pipeline.**

Before any pose production begins, FLUX 2 Pro generates hero frames. Boba reviews and approves:

1. Generate Kael hero frame (FLUX 2 Pro, 1024×1024)
2. **Boba reviews:** silhouette, palette, proportions, costume, expression
3. If FAIL → iterate prompt, regenerate. Budget: up to 10 attempts.
4. If PASS → downscale to 512×512 (nearest-neighbor). This becomes the `input_image` for all Kael Kontext Pro calls.
5. Repeat for Rhena.
6. Repeat for P2 palettes (if producing alternate costumes).

**Hero frame approval criteria:**
- Character is instantly recognizable at 64×64 silhouette
- All palette colors within ±5% of target hex
- Costume matches Section 4 description exactly (wraps, belt, scars, etc.)
- Pose is clean idle stance
- No artifacts, no background bleed

### 7.4 Batch Strategy

**Batch size:** Generate one full pose (all frames) per batch.

**Process:**
1. Load hero frame as base64 `input_image`
2. Generate frame 1 of pose with base seed
3. **Review immediately** — silhouette, colors, pose accuracy
4. If frame 1 passes → generate remaining frames (increment seed per frame)
5. Review full sequence at 60 FPS in Godot AnimationPlayer
6. Regenerate any outlier frames (wrong proportions, missing details)
7. Move to next pose

**Rate limit management (Kontext Pro at 30 req/min):**
- 1 frame every 2 seconds is sustainable
- 8-frame pose = ~16 seconds generation
- 12-frame pose = ~24 seconds generation
- Include 5-second review pause between frames = realistic pace of ~1 pose per minute
